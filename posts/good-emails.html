Being able to communicate well is second only to your ability to write good code as a developer. It takes a lot of work to get right, and getting it right is really important. Here are my tips.

1. Be nice

Sounds easy, right? It is, but I'm amazed at how often people are angry in their emails. A good friend once told me "Don't document negativity".

Emails are forever. Don't let anger, which is usually a temporary emotion, leak into your emails. If you need to, go take a walk and cool down before you hit send.

You may want to side-step this rule by writing polite emails which are insults in disguise. For example, "Thank you for finally resolving this", or "I think you've helped all that you can!". Don't do this.

2. Be brief

People skim their emails, so be terse. If a long explanation is warranted, consider putting a one line summary at the top.

Write an email draft and then refine and trim it until the email contains just the bare essentials.

3. If you expect action, make it painfully clear

Here's a good example of a concise email:

From: Me
Subject: Feature-x ready for testing
Dear team,
We have developed feature-x and it is ready for testing. Feature-x allows our software to import Word files. Jim, can you check out brach-feature-x and test it? Let us know if it contains bugs or if it is working as expected.

Bonus points if you put the request in the first line. More bonus points if the email is one line.

As a corollary to this rule, if you don't expect anybody to take action, reconsider whether an email is needed at all.

However, email can be useful as a paper trail. If somebody says, "You never told me!", email proves them wrong instantly.

4. Read through your emails before you send them

You'll be amazed at how many mistakes you'll find. I generally keep re-reading an email until I get through it without making any changes

5. Replies should empower the sender

When you receive an email, it's usually because the sender is trying to accomplish some task, and needs your help. Here's an example email.

From: Customer
Dear development team,
We're receiving reports that the software is not working this morning. Please look into this right away.

What the customer is actually saying:

From: Customer
Dear development team,
My boss has gotten over 50 emails from pissed off users that our software is broken. He's breathing down my neck demanding answers. If this isn't fixed, somebody is going to get fired, and it won't be me. Help!

Here's a bad way to handle it

From: Developer
Dear Customer,
This problem might be occurring because a user sent too much data, or the operations team was performing an upgrade (Monday night is normally when upgrades occur, which we've repeatedly warned was a bad time). The upload module is very complex. It's possible that it's actually working fine, and the user doesn't understand how the software should behave. I've attached a copy of our user manual. We should probably discuss some of the edge cases for upload which weren't specified before.
Signed, Not-our-problem

Okay, that's probably over the top, but this is based on a true story. You haven't empowered the customer to do anything to help fix his/her situation. Here's a better way to handle that same situation:

From: Developer
Dear Customer,
We don't know why this is happening. We are looking into this and will get back to you this morning.
Signed, <name>
Desk: <number>

Saying "I don't know" is a perfectly cromulent response if it's honest. Be sure to leave an alternate way to contact you if this is the case. Here's an example of a good follow up:

From: Developer
Dear Customer,
We looked into the incident. At 7:30PM last night, our software was reporting two dots, where normally, there is only one dot. The root cause of this was a user uploading 400TB of data at 5:02PM that same day. We understand having two dots is not desirable, as it adversely affects how the app performs. At 9:25AM today, we logged into the server and manually reduced the number of dots to one. In the short term, we will periodically log into the server and check how many dots are present. We can prevent this from happening in the future by either restricting how much data is uploaded, or setting up a second server to handle the load. We can prepare this software change with the upcomming software release planned on the 31st, or prepare an emergency patch which can be deployed on Thursday at the earliest.
Signed, <name>

You don't have to apologize, but you can if you feel it's appropriate. The email is informative, direct and honest. With this type of response, your customer is now empowered to answer all the questions he/she is receiving.

You can be the greatest developer, but if you're not effective at communicating, you will likely be viewed as a poor developer. Take the time to write good emails, and it will pay in dividends.
