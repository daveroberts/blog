Uh oh, the Internet's down

How far can you get in your favorite programming language without using Google? Can you write "Hello World"?

"Certainly."

puts "Hello World"

What if Java is your favorite language?

"A little harder, but I think I can manage."

public class Main {
  public static void main(String[] args){
    System.out.println("Hello World")
  }
}

"Then run javac Main.java and then java Main."

Okay, let's move beyond Hello World. Can you read input from the user?

BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
String input = br.readLine();

Be sure to import java.io.* at the start of your file

Did you cheat and have to look that up? [1]

As a programmer, we cache a certain amount of our language / platform into memory, and for everything else we use Google. Off the top of my head, here are some basic things I usually have cached:

  • Basic IO
  • Declaring variables, functions, constants, classes
  • Assigning values, basic math, equality checks, calling functions, calling methods
  • Loops, if statements, flow control
  • Some string functions, arrays or collections, some collection functions

For everythig else, I hit up Google. [2]

Why caching matters

You may think, "Who cares? I have Google. I don't need to remember everything!" and you're right. However, I assert that you will be happier with the language / platform where you're able to have the most "stuff" cached in your brain.

This isn't just an issue that beginners experience. Nobody has a big enough brain to hold a whole language. If a language is so complex that you can't fit basic operations in your brain, your brain may tell you, "Why bother using this language?" As a language, API or platform designer, you can take advantage of this fact by modeling your language, API or platform to be like an already popular one. This way, even new users will have some subset of your features pre-cached

Rails success

I believe a key factor in Rails success was how easy it was for a developer to cache the API in their brain. David Heinemeier Hansson put out a video where he built a blog in 15 minutes from memory. Yehuda Katz made a video doing the same thing, but didn't use any pre-existing code. I too am able to create a Rails website without once looking at the API.

This isn't because I'm a great programmer. This is a reflection of how great the Rails API is. Everything from downloading Rails to setting up the database can be done easily from memory.

Boilerplate

Getting started with a new platform should be easy. Here is one file from the first page of the official Angular tutorial:

import { NgModule }      from '@angular/core';
import { BrowserModule } from '@angular/platform-browser';
import { FormsModule }   from '@angular/forms'; // <-- NgModel lives here

import { AppComponent }  from './app.component';

@NgModule({
  imports: [
    BrowserModule,
    FormsModule // <-- import the FormsModule before binding with [(ngModel)]
  ],
  declarations: [
    AppComponent
  ],
  bootstrap: [ AppComponent ]
})
export class AppModule { }

I realize that this isn't a 'Hello World', and a bare bones Angular app can problem be smaller, but looking at this file, there's a lot for a beginner to take in. Compare this with Angular 1:

<input type="text" ng-model="name" />
<h1>Hello {{ name }}</h1>

You already understand everything in the example. You don't need to look anything up. I realize that there are edge cases, gotchas, and they had to rewrite the application for reasons, but looking at these two examples, it seems that for everything gained, the Angular team lost simplicity in the transition.

Clone my starter project

Lots of projects have a getting started guide where the first step is to clone a "starter pack" from GitHub. While it's important to facilitate getting started, this can also hide a lot of complexity.[3]

Having a quick start is great, but if I have to use the quick start every time, then getting started is too hard to bother remembering.

Why this matters

If you're designing a Language, platform, or API. I believe a key factor of its success is determined by how simple it is to use. You might have a platform / language / API that is successful for other reasons, despite being hard to use, but you're going against the grain.
